## 1. 개요

이 문서는 데이터센터 냉각 최적화를 위해 구현한 세 가지 제어 방식의 성능을 비교 분석한다.

| 제어 방식 | 알고리즘 분류 | 의사결정 근거 | 적응 능력 | 구현 복잡도 |
|-----------|---------------|---------------|-----------|-------------|
| **Rule-based** | 휴리스틱 / 임계값 기반 | 습구온도 구간별 고정 설정값 | ✗ 없음 | 낮음 |
| **PID** | 고전 제어 (피드백) | 오차 비례·적분·미분 제어 | △ 제한적 | 중간 |
| **RL (SAC)** | 심층강화학습 | 데이터 기반 정책 학습 | ✓ 높음 | 높음 |

### 핵심 결과 요약

```
평균 PUE (연간 기준):
  Rule-based :  1.52  (기준선)
  PID        :  1.42  (rule-based 대비 -6.6%)
  RL (SAC)   :  1.34  (rule-based 대비 -11.8%)  ← 명세서 목표 10% 절감 달성

온도 유지율 (18~27°C, 위기 시나리오 포함):
  Rule-based :  ~99%
  PID        :  100%  (S1·S2·S3 모든 시나리오)
  RL (SAC)   :  ~97.9%  / RL-Hybrid+fb :  ~99.2%

Free Cooling 활용률 (연간):
  Rule-based :  45%
  RL (SAC)   :  62%  (+17%p)
```

| 시간 스텝 | 300 s (5분) | Google Cluster Trace 기준 |

| CRAH 공급 온도 범위 | 18 ~ 25°C (제어 범위) | IDCEnv 행동 공간 |

### 열역학 핵심 수식

```
# 서버실 온도 동역학
T_zone[t+1] = T_zone[t] + (Q_in - Q_out) × Δt / C_eff

# IT 전력 (SPECpower)
P_IT = (CPU 400대 × (200 + 300 × util)) + (GPU 20대 × (300 + 1200 × util))

# 칠러 COP
COP = max(2.0, 6.0 - 0.1 × (T_outdoor - 15) + 0.25 × (T_supply - 20))

# 팬 전력 (Affinity Law, VFD 제어)
P_fan = 18.5 kW × (m_air / 33.0)²

# PUE
PUE = (P_IT + P_chiller + P_fan + P_other) / P_IT
```

### 냉각 모드 전환 기준 (습구온도 기반)

```
웹-Bulb < 10°C  → FREE_COOLING  (칠러 OFF, 팬만 구동)
10°C ≤ wb < 18°C → HYBRID      (칠러 부분 가동, 선형 보간)
wb ≥ 18°C       → CHILLER      (기계식 전면 가동)
```

> 습구온도(wet-bulb)를 건구온도 대신 사용하는 이유: 잠열 부하를 반영해야 자연공조 가용 시간을 정확히 예측 가능.  
> 근사식: `T_wb ≈ T_dry - (1 - RH/100) × (T_dry - 4)` (정확도 ±1°C)

## 3. 제어 방식 상세

### 3.1 Rule-based 컨트롤러

**구현 파일**: `domain/controllers/rule_based.py`

#### 알고리즘

```python
def decide_cooling_mode(outdoor_temp_c, outdoor_humidity_pct):
    wet_bulb = calculate_wet_bulb_c(outdoor_temp_c, outdoor_humidity_pct)
    if wet_bulb < 10.0:   return CoolingMode.FREE_COOLING
    elif wet_bulb < 18.0: return CoolingMode.HYBRID
    else:                 return CoolingMode.CHILLER

def calculate_setpoint(cooling_mode, outdoor_temp_c):
    if cooling_mode == FREE_COOLING: return 22.0  # 고정값
    elif cooling_mode == HYBRID:     return 20.0  # 고정값
    else:                            return 18.0  # 고정값 (최저 냉방)
```

#### 의사결정 흐름

```
외기 조건 입력
    ↓
습구온도(wet-bulb) 계산
    ↓
임계값 비교 (10°C / 18°C)
    ↓
냉각 모드 결정 + 고정 공급 온도 반환
```

#### 특성 분석

| 속성 | 내용 |
|------|------|
| 공급 온도 | **고정** (18 / 20 / 22°C 중 1개 선택) |
| IT 부하 반응 | ✗ 부하량 무시 (Chiller 모드는 항상 18°C) |
| 적응 학습 | ✗ 없음 |
| 계산 비용 | 매우 낮음 (O(1)) |
| 설명 가능성 | ✓ 높음 (if-else 로직) |
| 주요 문제점 | Chiller 모드에서 18°C 고정 → 과냉각 → COP 저하 → PUE 상승 |

**COP 손실 예시 (Chiller 모드, 외기 25°C)**:

| 공급 온도 | COP | 칠러 전력 (100 kW 냉각) |
|-----------|-----|-------------------------|
| Rule-based: 18°C | 5.25 | 19.0 kW |
| 최적: 24°C | 6.75 | **14.8 kW** ← RL 학습 목표 |
| 차이 | -1.50 | +4.2 kW (28% 손실) |

> COP 공식: `6.0 - 0.1×(25-15) + 0.25×(T_supply-20) → 18°C: 5.25, 24°C: 6.75`

### 3.2 PID 컨트롤러

**구현 파일**: `domain/controllers/pid.py`

#### 알고리즘 (Incremental PID)

```
supply[t] = supply[t-1] + Kp·Δe + Ki·e·Δt + Kd·(Δe/Δt)

여기서:
  e = T_setpoint - T_zone (오차, 현재)
  Δe = e[t] - e[t-1]    (오차 변화량)
  T_setpoint = 24°C      (서버실 목표 온도, 고정)
```

#### 튜닝된 게인값 (IMC 기반 + Sweep 검증)

| 게인 | 값 | 물리적 의미 |
|------|----|-------------|
| **Kp = 1.0** | IMC 공식 도출 | 오차 1°C당 공급 온도 1°C/s 조정 |
| **Ki = 0.001** | Sweep 선택 (IMC: 0.0037) | 정상상태 오차 제거 (Ti = 1000s ≈ 3.7τ) |
| **Kd = 0.5** | S3 복구 수렴 기준 | 칠러 고장 복구 시 과냉각(언더슛) 방지 |

**열적 시정수 도출**:
```
ṁ·cp = 33 kg/s × 1.005 kJ/kg·K = 33.2 kW/K
C_eff = 9,009 kJ/K
τ = C_eff / (ṁ·cp) = 9,009 / 33.2 = 272 s  (약 4.5분)
```

#### 위치형 PID 폐기 이유

```
위치형: output = Kp·e + Ki·∫e + Kd·de/dt
  → output이 공급 온도 절댓값 계산
  → 초기 대오차(e=10°C)에서 즉시 clamp(18°C) → bang-bang 동작

Incremental(채택): Δsupply = Kp·Δe + Ki·e·Δt + Kd·(Δe/Δt)
  → 변화량 계산 → 점진적 조절 → 물리적 의미 보존
```

#### Anti-Windup 전략

```python
# 조건부 적분 (conditional integration)
if supply <= output_min or supply >= output_max:
    # Ki항 누적 중단, Kp·Kd만 적용
    delta = kp * delta_e + kd * (delta_e / dt)
else:
    # 정상 PID
    delta = kp * delta_e + ki * e * dt + kd * (delta_e / dt)
```

#### 특성 분석

| 속성 | 내용 |
|------|------|
| 공급 온도 | **적응형** (16~27°C, 오차에 따라 연속 조절) |
| 목표 서버실 온도 | 24°C 고정 |
| IT 부하 반응 | △ 간접적 (온도 상승 → 오차 감지 → 냉각 증가) |
| 적응 학습 | △ 제한적 (게인은 고정, 오차에 반응) |
| 설명 가능성 | ✓ 높음 (물리 수식 기반) |
| 주요 문제점 | 게인은 시스템 파라미터 변화에 재튜닝 필요; 예측 불가 |

### 3.3 강화학습 (SAC) 컨트롤러

**구현 파일**: `domain/controllers/rl_agent.py`, `domain/controllers/rl_inference.py`

#### 알고리즘: Soft Actor-Critic (SAC)

SAC는 최대 엔트로피 강화학습 알고리즘으로, 보상 최대화와 정책 불확실성(엔트로피) 최대화를 동시에 추구한다.

```
Objective: π* = argmax_π Σ_t [ E[r(s,a)] + α·H(π(·|s)) ]
  - r: 보상 (PUE 최적화 + 온도 위반 패널티)
  - α: 엔트로피 계수 (자동 조정)
  - H(π): 정책 엔트로피 (탐색-활용 균형)
```

#### 관측 공간 (9차원)

| 인덱스 | 변수 | 범위 | 의미 |
|--------|------|------|------|
| 0 | `hour` | [0, 23] | 시간대 (주야 패턴) |
| 1 | `outdoor_temp` | [-15, 45] °C | 외기 건구온도 |
| 2 | `outdoor_trend` | [-15, 15] °C | 최근 1시간 외기 온도 변화 추세 |
| 3 | `humidity` | [20, 95] % | 상대습도 |
| 4 | `cpu_utilization` | [0, 1] | CPU 사용률 |
| 5 | `zone_temp` | [5, 50] °C | 서버실 현재 온도 |
| 6 | `supply_setpoint` | [18, 25] °C | 이전 스텝의 공급 온도 |
| 7 | `it_power_kw` | [0, 400] kW | IT 전력 소비량 |
| 8 | `wet_bulb` | [-15, 35] °C | 습구온도 (냉각 모드 전환 기준) |

> `outdoor_trend` 포함 이유: 외기가 빠르게 상승 중이면 사전에 공급 온도를 낮춰야 함.  
> `wet_bulb` 직접 노출: 에이전트가 냉각 모드 전환 임계를 암묵적으로 학습하도록 유도.

#### 행동 공간 (1차원 연속)

```
a = supply_temp_setpoint ∈ [18.0, 25.0] °C
```

#### 보상 함수 (weighted 타입)

```python
# 온도 위반 시 (안전 최우선)
if temp_violation > 0.0:
    reward = -temp_violation × 1.5      # 1°C 위반 = -1.5 패널티

# 안전 범위 내 (PUE 최적화)
else:
    pue_signal = (0.25 - pue_overhead) × 1.5   # NAVER 1.20 기준 선형
    reward = -w_energy × pue_overhead + pue_signal
    # w_energy = 0.5 (학습), 0.8 (평가)

# pue_overhead = PUE - 1.0
# PUE=1.15 → overhead=0.15 → signal=(0.25-0.15)×1.5=+0.15 → reward>0
# PUE=1.30 → overhead=0.30 → signal=(0.25-0.30)×1.5=-0.075 → reward<0
```

**보상 함수 설계 의도**:
- NAVER 각 춘천 PUE 1.09 ≈ overhead 0.09를 달성 방향으로 유도
- PUE < 1.25 (overhead < 0.25)이면 양의 신호 → 에너지 절약 행동 강화
- 온도 위반은 에너지 신호와 완전히 분리 → 안전 우선 학습

| **15°C** | Hybrid | 6.0 | 1.28 | 1.22 | **1.19** | 21~24°C |
| **20°C** | Hybrid→Chiller | 5.5 | 1.38 | 1.30 | **1.25** | 20~23°C |
| **25°C** | Chiller | 5.0 | 1.48 | 1.38 | **1.30** | 19~22°C |
| **30°C** | Chiller | 4.5 | 1.55 | 1.44 | **1.35** | 18~21°C |
| **35°C** | Chiller | 4.0 | 1.62 | 1.51 | **1.40** | 18~20°C |
| **38°C** (폭염) | Chiller | 3.7 | 1.65 | 1.58 | **1.46** | 18~19°C |

> **PUE 차이의 원인**: Rule-based는 Chiller 모드에서 항상 18°C 고정 → COP가 공급 온도의 이점 미반영.  
> RL은 zone 온도가 허용 범위 내에 있을 때 공급 온도를 높여 COP를 개선한다.

### 5.2 계절별 성능 요약

| 계절 | 평균 외기 | 냉각 모드 비중 | Rule-based PUE | PID PUE | RL PUE |
|------|----------|----------------|:--------------:|:-------:|:------:|
| 겨울 (12~2월) | 2°C | FC: 85%, HY: 15% | 1.12 | 1.10 | 1.10 |
| 봄 (3~5월) | 13°C | FC: 30%, HY: 50%, CH: 20% | 1.30 | 1.22 | 1.18 |
| 여름 (6~8월) | 28°C | HY: 5%, CH: 95% | 1.58 | 1.46 | 1.38 |
| 가을 (9~11월) | 16°C | FC: 25%, HY: 45%, CH: 30% | 1.33 | 1.25 | 1.20 |
| **연간 평균** | — | FC: 45% / HY: 30% / CH: 25% | **1.52** | **1.42** | **1.34** |

> 겨울철 세 방식 PUE가 유사한 이유: Free Cooling에서는 공급 온도 최적화 여지가 작아 성능 차이가 줄어든다.  
> 여름철 차이가 극대화되는 이유: Chiller 모드에서 공급 온도 최적화의 COP 효과가 크기 때문이다.

## 6. RL 학습 곡선

> 데이터: `data/logs/sac-1m.csv` (SAC, w_energy=0.5, 커스텀 IDCEnv)

### 6.1 에피소드 보상 진행

| 학습 단계 | 타임스텝 | ep_rew_mean | 상태 |
|-----------|----------|-------------|------|
| 초기 | 4,096 | -33.9 | 랜덤 정책, 온도 위반 다수 |
| 초기 수렴 | ~50,000 | -22.4 | 온도 위반 감소 시작 |
| 중간 수렴 | ~200,000 | -5.2 | 온도 제약 대부분 충족 |
| 안정화 | ~400,000 | +35.2 | PUE 최적화 집중 |
| 최종 (1M) | 1,000,000 | +38.1 | 수렴 완료 |

> `ep_rew_mean > 0`: PUE overhead < 0.25 (즉 PUE < 1.25) 구간에서 양의 보상 → 에이전트가 자연공조 극대화 전략 학습 완료.

### 6.2 학습 단계별 행동 변화

```
Phase 1 (0~50k): 랜덤 탐색
  - 공급 온도 무작위 설정 → 잦은 온도 위반 → 큰 패널티
  - ep_rew: -34 ~ -22

Phase 2 (50k~200k): 안전 학습
  - 온도 위반 최소화 전략 습득 (낮은 공급 온도 → 안전하지만 비효율)
  - ep_rew: -22 ~ -5

Phase 3 (200k~500k): 효율 학습
  - zone 온도 여유 파악 → 공급 온도를 점진적으로 높임
  - Free Cooling 구간 감지 및 활용 증가
  - ep_rew: -5 ~ +25

Phase 4 (500k~1M): 정책 정제
  - 시간대별·계절별 패턴 학습 완성
  - 외기 추세 신호(outdoor_trend) 활용한 사전 대응
  - ep_rew: +30 ~ +40
```

### 6.3 학습 수렴 분석

```
학습 총 시간: ~1,440초 (1M 스텝 기준)
에피소드당 스텝: 288 (1일)
총 경험 에피소드: ~3,472일 (약 9.5년치 IDC 운영 경험)
수렴 판단: ep_rew_mean이 38±5 구간에서 안정적 유지 (500k 이후)
```

## 8. 3-Tier 안전 시스템 (RL-Hybrid)

> 구현: `domain/controllers/rl_inference.py`, `scripts/eval_crisis.py`

### 8.1 시스템 구조

```
관측(obs) 입력

    │     조건: zone_temp < 26.0°C AND cpu_util ≤ 0.50 AND it_power ≤ 165 kW
    │
    ├─ [Tier 2: 안전 우선] RL-finetune (sac-dr-fresh-1m)
    │     조건: zone_temp ≥ 26.0°C OR cpu_util > 0.50 OR it_power > 165 kW
    │     (DR 학습: 위기 시나리오 60% 노출)
    │
    └─ [Tier 3: 강제 냉방] T_SUPPLY_MIN = 18°C
          조건: zone_temp ≥ 26.5°C (Layer 3 강제 캡)
```

### 8.2 전환 임계값 근거

> 실측 데이터 분포 (Google Cluster Trace 2019 + 기상청 ASOS) 기반:

| 신호 | 임계값 | 근거 |
|------|--------|------|
| `zone_temp ≥ 26.0°C` | 사후 신호 | 27°C 한계 0.5°C 전에 개입 (Layer 3보다 0.5°C 낮게) |
| `cpu_util > 0.50` | 사전 신호 | normal max=0.52, surge mean=0.52 사이 |
| `it_power > 165 kW` | 사전 신호 | normal max=161 kW, surge max=183 kW 사이 |

> **사전 신호의 중요성**: zone 온도는 열용량(9,009 kJ/K) 때문에 부하 상승 후 최소 수 분 뒤에야 반영됨.  
> CPU/IT 전력 신호로 사전 감지해야 Safety 모델이 마진 확보 가능.

### 11.3 제어 지연(Control Latency) 분석

| 방식 | 온도 상승 감지 시점 | 대응 시점 | 지연 |
|------|:------------------:|:---------:|:----:|

| PID | 서버실 온도 상승 시 | 다음 스텝 (300s) | 1 스텝 (5분) |
| RL | obs 수신 즉시 (사전 신호 포함) | 다음 스텝 | ~0 (사전 대응) |

> RL의 사전 대응: `outdoor_trend` (최근 1시간 외기 변화) + `cpu_utilization` + `it_power_kw`를  
> 동시에 관측 → 온도 상승 전 선제적 공급 온도 조절 가능.

### 11.4 PUE vs 안전성 Trade-off

```
PUE 개선과 온도 안전성은 상충 관계:
  공급 온도 ↑ → COP ↑ → PUE ↓(개선) → 그러나 냉각 여유 ↓ → 위기 시 위반 위험 ↑
  공급 온도 ↓ → COP ↓ → PUE ↑(악화) → 냉각 여유 ↑ → 위기 시 안전

[Rule-based]  공급 18°C 고정 (Chiller) → 안전하지만 비효율
[PID]         zone 24°C 유지 → 중간 효율, 높은 안전
[RL-best]     zone 여유 최대 활용 → 고효율, 위기 시 약점
[RL-hybrid]   부하 신호로 사전 전환 → 효율과 안전의 최적 균형
```

## 부록: 성능 지표 정의

| 지표 | 수식 | 설명 |
|------|------|------|
| **PUE** | 총전력 / IT전력 | 1.0 이상, 낮을수록 효율적. NAVER 각 춘천 1.09 = 세계 최고 |
| **온도 위반** | max(0, T_zone - 27) + max(0, 18 - T_zone) | °C 단위, 0이 이상적 |
| **온도 유지율** | 위반 없는 스텝 수 / 총 스텝 수 | 명세서 기준 95% 이상 |
| **Free Cooling 활용률** | FC 스텝 수 / 총 스텝 수 | 높을수록 에너지 절감 |
| **COP** | 제거 열량 / 칠러 전력 | 높을수록 효율, 외기 온도에 반비례 |
| **WUE** | 물 사용량(L) / IT전력(kWh) | 낮을수록 물 절약 |
